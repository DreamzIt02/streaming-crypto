# Stream V3

The refactored pipeline would look if we replace `fatal_tx<StreamError>` with a **`monitor_tx<Result<Telemetry, StreamError>>`**:

---

## 🔹 Channel Layout (Refactored)

```bash
Reader → Compression → Encryption → Writer
    │       │           │               │
    └──-----└──---------└──-----------──┘
                    ↓
    monitor_tx<Result<Telemetry, StreamError>>
```

- **Data path (parallel):**
  - Reader threads push clean chunks → Compression workers compress → Encryption workers encrypt → Writer writes.
  - All data channels carry **only valid payloads** (`EncryptSegmentInput` → `EncryptedSegment`).
  - No `Result` wrappers in the data path.

- **Monitor path (centralized):**
  - Every worker sends telemetry events (`Ok(Telemetry)`) into `monitor_tx`.
  - On error, workers send `Err(StreamError)` into `monitor_tx`.
  - Monitor thread consumes this channel, aggregates telemetry, and cancels pipeline if needed.

---

## 🔹 Monitor Thread Logic

```rust
scope.spawn(move || {
    for msg in monitor_rx.iter() {
        match msg {
            Ok(t) => {
                // Aggregate telemetry
                telemetry.merge(t);
            }
            Err(e) => {
                error!("[MONITOR] fatal error: {e}");
                *fatal_error_m.lock().unwrap() = Some(e);

                // Signal cancellation
                cancel_m.cancelled.store(true, Ordering::Relaxed);
                cancel_m.senders.lock().unwrap().clear();
                cancel_m.receivers.lock().unwrap().clear();
                break;
            }
        }
    }
    debug!("[MONITOR] pipeline completed successfully");
});
```

---

## 🔹 Benefits of This Design

- ✅ **No Adapter thread**: Compression workers send clean segments downstream, errors go directly to monitor.
- ✅ **Unified telemetry + error handling**: One channel, one consumer, one place to aggregate and react.
- ✅ **Parallel pipeline preserved**: No sequential bottleneck in the data path.
- ✅ **Cleaner contracts**: Encryption workers only deal with valid segments, monitor thread handles all errors and telemetry.

---

## 🔹 Considerations

- **Telemetry granularity**: Define a `Telemetry` struct that can represent stage times, byte counts, counters, etc. Workers emit events per segment or per batch.
- **Throughput**: Monitor thread must handle potentially high-frequency telemetry events. Usually fine, but we may want batching (workers send aggregated telemetry periodically).
- **Cancellation**: Errors still trigger cancellation immediately, same as before.

---

👉 In short: by switching to `monitor_tx<Result<Telemetry, StreamError>>`, we unify monitoring and error handling, eliminate the Adapter thread, and keep the pipeline fully parallel.

### 🔹 Implementation

```rust
impl PipelineMonitor for Monitor {
    #[inline]
    fn report_error(&self, err: StreamError) {
        match self.monitor_tx.send(Err(err)) {
            Ok(()) => eprintln!("[MONITOR] Error sent"),
            Err(e) => {
                eprintln!("[MONITOR] ❌ FAILED to send error!");
                eprintln!("[MONITOR] ❌ Lost error: {}", e.0);
                panic!("Fatal error could not be reported: {}", e.0);
            }
        }
        self.cancelled.store(true, Ordering::Relaxed);
        self.senders.lock().unwrap().clear();
        self.receivers.lock().unwrap().clear();
    }

    #[inline]
    fn report_telemetry(&self, t: TelemetryEvent) {
        let _ = self.monitor_tx.send(Ok(t));
    }

    #[inline]
    fn is_cancelled(&self) -> bool {
        self.cancelled.load(Ordering::Relaxed)
    }
}
```

### 🔹 Constructor & Finish

```rust
impl Monitor {
    pub fn new(
        senders: Vec<Box<dyn Any + Send>>,
        receivers: Vec<Box<dyn Any + Send>>,
    ) -> (Self, Receiver<Result<TelemetryEvent, StreamError>>) {
        let (monitor_tx, monitor_rx) = crossbeam::channel::unbounded();
        let cancelled = Arc::new(AtomicBool::new(false));

        (
            Self {
                cancelled,
                monitor_tx,
                senders: Arc::new(Mutex::new(senders)),
                receivers: Arc::new(Mutex::new(receivers)),
            },
            monitor_rx,
        )
    }

    #[inline]
    pub fn finish(self) {
        self.senders.lock().unwrap().clear();
        self.receivers.lock().unwrap().clear();
    }
}
```

### 🔹 TelemetryEvent Design

```rust
#[derive(Debug, Clone)]
pub enum TelemetryEvent {
    StageSnapshot {
        stage_times: StageTimes,
        counters: TelemetryCounters,
    },
    PipelineFinished {
        final_stage_times: StageTimes,
        final_counters: TelemetryCounters,
    },
}
```

- **StageSnapshot**: emitted by each worker after finishing a segment or batch.  
  - Contains `StageTimes` (with durations measured inside the worker).  
  - Contains `TelemetryCounters` (bytes processed, frames counted, etc.).  
- **PipelineFinished**: emitted once at the end, so the monitor knows the pipeline completed successfully.

---

### 🔹 Worker Responsibility

- Each worker owns a `TelemetryTimer` locally.
- At the end of its work on a segment:
  1. Call `timer.stage_times.add(stage, duration)`.
  2. Build a `TelemetryEvent::StageSnapshot`.
  3. Send it via `monitor.telemetry(event)`.

Example inside a compression worker:

```rust
let start = Instant::now();
let compressed = compress(chunk)?;
let duration = start.elapsed();

let mut stage_times = StageTimes::default();
stage_times.times.insert(Stage::Compress, duration);

let counters = TelemetryCounters {
    bytes_plaintext: chunk.len() as u64,
    bytes_compressed: compressed.len() as u64,
    ..Default::default()
};

monitor.telemetry(
    TelemetryEvent::StageSnapshot { stage_times, counters }
);
```

---

### 🔹 Monitor Thread Aggregation

The monitor thread merges snapshots into the global telemetry state:

```rust
for msg in monitor_rx.iter() {
    match msg {
        Ok(TelemetryEvent::StageSnapshot { stage_times, counters }) => {
            global_timer.stage_times.merge(&stage_times);
            global_counters.merge(&counters);
        }
        Ok(TelemetryEvent::PipelineFinished { final_stage_times, final_counters }) => {
            global_timer.stage_times.merge(&final_stage_times);
            global_counters.merge(&final_counters);
        }
        Err(err) => {
            error!("[MONITOR] Fatal error: {}", err);
            cancel.fatal(err);
            break;
        }
    }
}
```

### Python → Rust zero‑copy via PyO3

On the Rust side:

```rust
use pyo3::prelude::*;
use pyo3::types::PyBytes;
use std::slice;

#[pyfunction]
fn process_buffer(py: Python<'_>, py_bytes: &PyBytes) {
    // Get raw pointer and length from Python bytes
    let ptr = py_bytes.as_bytes().as_ptr();
    let len = py_bytes.as_bytes().len();

    // SAFETY: Python must keep the buffer alive while Rust uses it
    let data: &[u8] = unsafe { slice::from_raw_parts(ptr, len) };

    // Now we can parallelize reads directly on `data` without copying
    parallel_read(data);
}
```

Or, if we want to accept a `memoryview` (better for large buffers):

```rust
use pyo3::prelude::*;
use pyo3::types::PyMemoryView;
use std::slice;

#[pyfunction]
fn process_memview(memview: &PyMemoryView) {
    let buf = unsafe { memview.as_slice::<u8>().unwrap() };
    // buf: &[u8] borrowed directly from Python
    parallel_read(buf);
}
```

### Parallel read in Rust (true zero‑copy)

```rust
use crossbeam::scope;

fn parallel_read(data: &[u8]) {
    let len = data.len();
    let chunk_size = 1024 * 1024; // 1 MB chunks
    let workers = 4;

    scope(|s| {
        for worker_id in 0..workers {
            let data = data; // borrow only
            s.spawn(move |_| {
                let mut seg = worker_id;
                while seg * chunk_size < len {
                    let start = seg * chunk_size;
                    let end = ((seg + 1) * chunk_size).min(len);

                    let slice = &data[start..end]; // ✅ zero‑copy slice

                    // process slice here
                    do_work(slice);

                    seg += workers;
                }
            });
        }
    }).unwrap();
}

fn do_work(chunk: &[u8]) {
    // encryption/compression/etc.
}
```

### ⚠️ Safety notes

- Python must **not free or mutate** the buffer while Rust is using it.  
- We must ensure Rust threads finish before Python drops the object.  
- This is why `PyBuffer`/`PyMemoryView` is safer than raw `PyBytes`: it pins the memory until we release it.

---

🔑 With this setup:

- No `Arc::from` copy.  
- No `Bytes::copy_from_slice` per chunk.  
- Rust threads slice directly into Python’s memory.
